# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Context & Assumptions (read first)

This is the **first UI-bearing phase of a greenfield project** — no code, no `components.json`, no `CONTEXT.md` exists yet (`/gsd-discuss-phase` was explicitly skipped by the founder to define design first). This contract therefore establishes the **entire visual identity** that Phases 2–5 inherit; it is intentionally more foundational than a typical phase UI-SPEC.

Per founder instruction, this spec makes reasonable, well-justified design decisions rather than blocking on unanswered questions. Assumptions the founder should sanity-check before/during execution:

1. **Dark-mode-only in v1** (no light theme, no toggle). Chosen because (a) it reads as distinctly premium/"own identity" rather than generic white SaaS dashboard — directly serving DESIGN-01, (b) it fits a "stadium lights / high-stakes duel at night" brand mood appropriate for sports betting, (c) it halves the design/QA surface for a founder who has prioritized visual polish over feature breadth. Revisit if user testing shows a strong light-mode preference.
2. **Portuguese-only copy, imperative/infinitive verb forms** ("Criar conta", "Depositar") to sidestep the tu/você formality choice, which is unresolved for the Mozambican market. Flag to founder if a house style (tu vs. você) is later decided — copy would need a pass.
3. **shadcn/ui is the assumed component tool** (matches STACK.md). Since the app doesn't exist yet, there is no live `components.json` to read — the exact file this contract expects the executor to create is given verbatim in "Design System" below, so Phase 1 execution can run `npx shadcn init` non-interactively against it rather than needing a founder round-trip through ui.shadcn.com/create.
4. **Single variable font family** (not a display+body pairing) — deliberately chosen over two font files to control network payload for the "low-end Android, variable network" audience, while still avoiding the genericness of Inter/Roboto-everywhere.

If any of these are wrong, correct them here before Phase 1 planning locks — they cascade to every later phase.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing, form field padding |
| lg | 24px | Section padding, card padding |
| xl | 32px | Layout gaps between major blocks |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level top/bottom spacing |

Exceptions:
- **44px minimum tap target** for all icon-only buttons and the 18+ confirmation checkbox's tap area (mobile-first / low-end-device accessibility — 8px alone is not a reliable touch target on budget Android screens with imprecise touch digitizers).
- **12px** used once, for the gap between the phone/M-Pesa and e-Mola method toggle chips on the deposit screen (breaks the 4pt scale deliberately for a denser two-up chip row on narrow viewports; acceptable because it's not a spacing unit reused elsewhere).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 14px | 500 (medium) | 1.4 |
| Body | 16px | 500 (medium) | 1.5 |
| Heading | 20px | 700 (bold) | 1.2 |
| Display | 32px | 700 (bold) | 1.1 |

Notes:
- Only two weights exist in the whole type system: **500 (medium)** and **700 (bold)**. Regular/400 is deliberately excluded — on low-DPI budget Android screens, 400-weight body text renders too thin to read comfortably; 500 is the effective "regular" here.
- **Display** is reserved for monetary figures only: the "Saldo Disponível" hero number on the wallet home screen, and the deposit-amount input's live preview. All monetary values use `font-variant-numeric: tabular-nums` (lining, tabular figures) so amounts don't jitter/reflow as digits change during the balance count-up animation (see Motion section).
- **Heading** is for screen titles ("Carteira", "Histórico", "Criar conta") and card section titles ("Saldo Bloqueado", "Últimas transações").
- **Label** is for form labels, transaction-row secondary text (date, reason code), and helper/hint text under inputs.
- Font loaded via `next/font/google({ subsets: ['latin', 'latin-ext'], weight: ['500','700'], display: 'swap' })` — zero runtime request to Google Fonts (Next.js self-hosts at build time), which matters directly for the "variable network" constraint.

## Color

Dark-mode-only palette (no light theme in v1 — see Assumption 1). Values given as hex for readability; implement via Tailwind v4's OKLCH-native `@theme` block in `app/globals.css` (convert hex → OKLCH at implementation time, e.g. with the browser devtools color picker or `culori`).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0D0E12` ("Duelo Ink") | App background, page canvas |
| Secondary (30%) | `#1A1C24` ("Duelo Slate") | Cards, bottom nav, sheets/modals, input fields |
| Accent (10%) | `#F2B807` ("Duelo Gold") | Reserved for: primary CTA buttons (Criar conta / Entrar / Depositar / Confirmar depósito), active tab/nav indicator, input focus ring, the brand logotype accent stroke, and the "Saldo Disponível" numeral emphasis on the wallet home hero |
| Destructive | `#F0455B` | Form validation errors, error toasts/banners, failed-deposit state, logout confirmation button |

Accent reserved for: primary CTA buttons, active nav/tab indicator, focus rings, brand logo accent — **nowhere else**. Do not use gold for decorative accents, illustrations, or secondary buttons; if gold appears more than once per screen outside this list, that's a violation of the 10% budget.

**Wallet-specific semantic colors** (required beyond the base 4 — WALLET-01 mandates available vs. locked balance be *visually* distinct, not just labeled):

| Role | Value | Usage |
|------|-------|-------|
| Success / positive | `#34D399` | Deposit-success confirmation state, credit-direction entries in transaction history (deposit, prize, refund), "Saldo Disponível" card accent line |
| Locked / escrow | `#8B8FF5` (soft indigo) | "Saldo Bloqueado" card accent line, locked-balance badge/icon — deliberately not red (locked funds are not an error state, just unavailable) |
| Border / divider (neutral) | `#2A2D38` | Card borders, input borders, list dividers — not counted in the 60/30/10 budget; a supporting neutral only |

This gives three balance states an instant, label-free visual signature: **available = green-tinted**, **locked = indigo-tinted**, **total = plain white/neutral text** — critical for WALLET-01's "shown separately" requirement to actually read as separate at a glance, not just as two numbers stacked in the same style.

## Motion & Micro-interactions

DESIGN-01 explicitly names micro-interactions as part of "own visual identity" — these are the concrete, buildable specs (Motion/`motion` package per STACK.md), scoped to stay light on low-end devices:

| Interaction | Spec |
|-------------|------|
| Balance count-up | When available balance changes (deposit confirmed), animate the Display-role numeral from old → new value over 500ms, ease-out, using `animate()` on a numeric motion value. Skip entirely (snap to final value) if `prefers-reduced-motion: reduce`. |
| Button press | `active:scale-[0.97]` + reduced shadow, spring back on release (`transition-transform duration-100`). Applies to all primary/CTA buttons — this is a CSS-only Tailwind transition, not a Motion-driven animation, to keep it free on low-end devices. |
| Input focus | Border color transitions to Accent Gold + a subtle `box-shadow` glow over 150ms ease. Signals "you're in a form field" without relying on color alone (accessibility). |
| Active tab/nav indicator | Sliding pill/underline via Motion's `layoutId` shared-layout animation, not an instant swap — this is the one place a genuine Motion (JS) animation is justified because the visual payoff (a "living" nav) is high and the element count is tiny (2–4 tabs). |
| Deposit success confirmation | Success-green checkmark icon scales in with a short spring (300ms, slight overshoot) inside the toast/sheet — the one deliberate "delight" moment in Phase 1, reserved for money successfully landing. |
| Loading states | Skeleton screens (CSS-only shimmer, no JS) for the wallet balance cards and transaction list on first load — never a spinner for content that has a known shape. |
| Refetch-on-reconnect | A thin top-of-screen progress bar (not a full-screen spinner) when TanStack Query refetches the wallet/transactions on window focus/reconnect — keeps the screen feeling "alive" over flaky mobile networks without blocking it. |

**Performance guardrail:** Motion (the JS library) is used only for the tab indicator and the deposit success checkmark in this phase — everything else (button press, input focus, skeleton shimmer) is plain CSS/Tailwind transitions. This keeps the client JS bundle small, which matters directly for the "low-end Android, variable network" constraint repeated throughout research.
