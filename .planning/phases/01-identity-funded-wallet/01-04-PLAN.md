<objective>
Close the identity slice with password recovery (AUTH-04): a user who forgot their password requests
a reset from the recover screen, receives an email link, sets a new password, and logs in with it.
Recovery is handled by Supabase Auth (short-lived single-use token, native reset), so no app-side
reset secret is invented. The forgot-password screen is enumeration-safe (identical response whether
or not the account exists).

Purpose: AUTH-04 completes the "register / stay logged in / reset password" identity requirement set
for Phase 1 and closes the account-recovery trust loop a money product depends on.
Output: A working recover → email → set-new-password → login flow.

> MVP slice: request-reset screen → email link → set-new-password screen → login with the new
> password. After this plan a real user who forgets their password can regain access — a capability
> they did not have before. Depends only on the auth foundation from Plan 01-01, so it runs in
> parallel with the wallet foundation (Plan 01-02).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → requestPasswordReset | Untrusted identifier; must not leak account existence |
| email link → auth callback | Recovery code arrives from the user's inbox and is exchanged for a session |
| recovery session → updatePassword | New password set under a short-lived recovery session |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-14 | Spoofing | password recovery flow | high | mitigate | Recovery uses Supabase's short-lived, single-use token delivered to the account email; updatePassword requires an active recovery session; no app-side guessable reset secret; new password re-validated |
| T-01-15 | Information Disclosure | forgot-password enumeration | medium | mitigate | requestPasswordReset returns the same generic confirmation for known and unknown identifiers; no distinct not-found error to the client |
| T-01-16 | Elevation of Privilege | updatePassword without recovery session | high | mitigate | updatePassword reads and requires a valid server-side session before calling updateUser; unauthenticated calls rejected |
</threat_model>
