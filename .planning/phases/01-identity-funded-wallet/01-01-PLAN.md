<objective>
Stand up the Duelo walking skeleton: a deployed Next.js 16 app with the full Duelo dark visual
identity, a real Supabase-backed register → login → session → authenticated dashboard flow, and
one real Postgres write (profiles insert) and read (dashboard). This is the thinnest end-to-end
slice that proves the entire stack (framework → server action → Supabase Auth → Postgres → themed
UI → Vercel deploy) is wired correctly before the wallet and deposit slices build on top.

Purpose: Establish the architectural backbone (SKELETON.md) and the visual identity (DESIGN-01)
that every later phase inherits, and deliver AUTH-01/02/03 as a working, beautiful capability.
Output: A running deployed app a real user can register into, stay logged into, and land on a
branded dashboard showing their name and a zero balance.
</objective>

## Phase Goal

**As a** new user in Mozambique, **I want to** register (confirming I'm 18+), stay logged in, and fund my wallet via M-Pesa or e-Mola, **so that** I can trust my money is safe and fully auditable before I ever place a bet.

> This plan delivers the identity + session + design-system slice of that goal. The wallet slice
> is Plan 01-02, the deposit slice Plan 01-03, and password recovery Plan 01-04.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server action | Untrusted registration/login input crosses here (phone, email, password, ageConfirmed) |
| server action → Supabase Auth / Postgres | Credential + profile persistence |
| npm registry → build | Third-party package code enters the app (supply chain) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Spoofing | lib/actions/auth.ts signIn / Supabase Auth | high | mitigate | Credentials hashed by Supabase Auth (never hand-rolled); Zod-validated input; rely on Supabase's built-in login rate limiting; no credential logging |
| T-01-02 | Information Disclosure | session cookie via @supabase/ssr | high | mitigate | httpOnly + secure + sameSite cookies (ssr defaults); middleware.ts refreshes/validates on every request; no token in client JS |
| T-01-03 | Elevation of Privilege | 18+ confirmation (AUTH-02) | medium | mitigate | ageConfirmed enforced server-side in registerUser (reject if false); age_confirmed_at persisted — client checkbox is never the sole gate (primary trust boundary) |
| T-01-04 | Tampering | phone→email identifier resolution in signIn | medium | mitigate | Identifier resolved against server-side profiles row; password always verified by Supabase; client cannot assert identity |
| T-01-08 | Information Disclosure | profiles table via PostgREST/RLS | high | mitigate | RLS owner-only (id = auth.uid()) select/update; inserts only via server action using service-role key server-side |
| T-01-SC | Tampering | npm/shadcn installs (supply chain) | high | mitigate | Blocking-human package legitimacy checkpoint (Task 1) verifies every package on npmjs.com before install; no [SUS]/[SLOP] packages permitted |
</threat_model>
