<objective>
Deliver Duelo's headline Phase 1 capability: a user funds their wallet via M-Pesa or e-Mola through
PaySuite, and — once PaySuite confirms via a signature-verified, idempotent webhook — their available
balance rises and the deposit appears in a full, auditable transaction history. Because the PaySuite
contract is LOW confidence (single-source, unverified), the integration is built defensively:
env-configurable endpoints/field names/signature header, constant-time HMAC verification, an
idempotency key written in the same DB transaction as the ledger credit, and a server-side status
poll as a fallback for a delayed webhook.

Purpose: PAY-01/02/03 (deposit + signed idempotent webhook) fund the wallet and establish the
reusable idempotent-webhook pattern Phase 4 withdrawals will inherit; WALLET-04 gives every user a
complete, reason-coded audit trail.
Output: A working deposit flow (initiate → pending → confirmed) that credits exactly once and a
transaction history screen.

> MVP slice order: thin deposit UI + initiate (user can start a deposit and see a pending state) →
> real credit via the verified idempotent webhook → transaction history refinement. After this plan
> a real user can put money into Duelo and see it, auditable — the core of the phase.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PaySuite → webhook route | Untrusted, internet-facing callback that moves money; the most exposed boundary in the phase |
| client → deposit initiate | User proposes an amount/method (must not be trusted as proof of payment) |
| app → PaySuite (outbound) | Bearer secret must stay server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-09 | Spoofing | app/api/webhooks/paysuite/route.ts | critical | mitigate | HMAC-SHA256 signature verified with crypto.timingSafeEqual over the RAW body BEFORE any processing; invalid → 401, credits nothing (tests/paysuite.signature.test.ts, paysuite.webhook.test.ts) |
| T-01-10 | Tampering | duplicate/replayed webhook delivery | critical | mitigate | Idempotency: request_id inserted into processed_webhook_events (PK, on-conflict-do-nothing) in the SAME tx as wallet_credit; replay is a no-op returning 200 — credited exactly once |
| T-01-11 | Tampering | client-side "payment success" redirect | high | mitigate | Wallet credited only by the verified webhook (or authenticated server-side status poll); client redirect is UX-only and never marks a deposit complete |
| T-01-12 | Tampering | client-supplied deposit amount | high | mitigate | Credited amount derived server-side from the PaySuite-confirmed payload matched to the stored PENDING deposit_intent; client amount only proposes the payment request, never the credit |
| T-01-13 | Information Disclosure | PAYSUITE_API_KEY / webhook secret | high | mitigate | Secrets server-only env (route handlers / server actions); never sent to the client bundle; .env.example documents without values |
</threat_model>
