<objective>
Build Duelo's financial bedrock and make it visible: an append-only double-entry ledger, a cached
balance row, and the atomic, row-locked PL/pgSQL mutation primitive (credit / hold / release) that
every money movement in the entire product will flow through — proven correct with a concurrency
test that fires simultaneous holds and confirms exactly one wins. Then surface it as the Duelo
wallet screen showing available, locked and total balance as three visually distinct values.

Purpose: WALLET-02 (auditable append-only ledger) and WALLET-03 (atomic, race-proof lock/unlock)
are the correctness foundation the money-first roadmap rests on; getting them right here means the
deposit slice (01-03) and Phase 2 escrow ride a proven primitive. WALLET-01 makes the three balance
states legible at a glance.
Output: A tested wallet-mutation primitive and a beautiful wallet screen showing a real (zero)
balance for the logged-in user.

> MVP slice: schema → atomic function (concurrency-tested) → balance read → wallet screen. After
> this plan a real user can open /wallet and see their three distinct balances — a capability they
> did not have after Plan 01-01. The hold/release functions have no UI consumer yet (no betting
> until Phase 2), but WALLET-03 requires the atomic primitive to exist and be proven now; the
> deposit credit path (01-03) exercises credit end-to-end.
</objective>

wallet.service.ts (lib/wallet/wallet.service.ts):
- credit(userId: string, amountMinor: number, ref: string, reason: LedgerReason): Promise<BalanceSnapshot>
- hold(userId: string, amountMinor: number, ref: string): Promise<BalanceSnapshot>
- release(userId: string, amountMinor: number, ref: string): Promise<BalanceSnapshot>
- getBalance(userId: string): Promise<BalanceSnapshot>

types (lib/wallet/types.ts):
- LedgerReason = 'deposit' | 'lock' | 'release' | 'payout' | 'commission' | 'refund' | 'withdrawal'
- LedgerAccount = 'available' | 'locked'
- BalanceSnapshot = { availableMinor: number; lockedMinor: number; totalMinor: number }

Postgres functions (called via supabase.rpc, defined in 0002_wallet_functions.sql):
- wallet_credit(p_user uuid, p_amount bigint, p_ref text, p_reason text) returns the new balance
- wallet_hold(p_user uuid, p_amount bigint, p_ref text) returns the new balance (raises on insufficient available)
- wallet_release(p_user uuid, p_amount bigint, p_ref text) returns the new balance
  Each: SELECT ... FOR UPDATE on the wallets row, validate, insert paired wallet_ledger rows,
  update wallets cache, all in one transaction. Lock order = ascending user_id for multi-row ops.

money (lib/money.ts): amounts are integer centavos everywhere; formatMT renders e.g. 1500 -> "15,00".

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| concurrent requests → wallets row | Two operations may race to mutate the same balance |
| any module → ledger tables | Only the SECURITY DEFINER functions may write; clients must not |
| client (PostgREST) → wallet_ledger / wallets | A user must never read another user's money |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-05 | Tampering | wallet_hold / wallet_credit balance mutation | critical | mitigate | SELECT ... FOR UPDATE row lock inside each PL/pgSQL function; validate-under-lock; consistent ascending user_id lock order; concurrency test (tests/wallet.concurrency.test.ts) proves exactly-one-success |
| T-01-06 | Tampering | direct balance write bypassing the ledger | high | mitigate | Client table writes revoked; all mutation only via SECURITY DEFINER functions; append-only ledger + cache updated in the same tx; CHECK constraints (amount > 0, balances >= 0) |
| T-01-07 | Repudiation | audit trail of money movements | medium | mitigate | Append-only wallet_ledger with reason + ref per entry (no updates/deletes); balance reconstructable from entries |
| T-01-08 | Information Disclosure | wallet_ledger / wallets via PostgREST | high | mitigate | RLS: SELECT only where user_id = auth.uid(); no client insert/update; processed_webhook_events denies all client access |
</threat_model>
